# Globe Physics in a Disc Costume

The author writes in his own About page:

> "All distances are unitless. `FE_RADIUS = 1`. No earth radius, no AU, no kilometres, no great-circle trigonometry."

Every clause of that sentence is contradicted by the source code. The model uses Earth radius, uses kilometres-equivalent (statute miles per degree of arc), and uses great-circle trigonometry as its primary tool for placing every body and every flight path. What follows are the load-bearing examples.

## Finding 1 — `MI_PER_DEG = 69.0936` is the Earth's radius in disguise

> **STD MODEL · MISLEADING**
> File: `js/demos/flightRoutes.js`, line 46

```js
// Convert linear mph to central-angle deg/h. The KML carries air
// speed in mph; this project displays everything in pure angle-and-
// time units, so per-waypoint speeds are normalised by the mean
// Earth great-circle, derived once and reused for every flight stat.
const MI_PER_DEG = 69.0936;
const mphToDegPerHour = (mph) => (mph == null || !isFinite(mph)) ? null : mph / MI_PER_DEG;
```

The number `69.0936` is the Earth's mean great-circle circumference (≈ 24 873 statute miles) divided by 360°. It is a globe constant, full stop — the Earth's radius `R` enters the model as `2πR / 360`, with the same numerical content as `R` itself. It is then used to convert real airline-recorded mph (from the bundled QF27/28 KMZ data) into the model's "central-angle deg/h" units.

The model further declares, in the same file:

> "this project stays in central-angle + time units and **never reports linear distance / speed**."

The reason this defensive perimeter exists is that any actual mph or km/h reading would be falsifiable against published airline operations. By converting the readings into deg/h (using the Earth's circumference) and then refusing to convert back, the model produces a number that *looks* unit-free while being arithmetically identical to the globe's prediction. The work is being done by the globe's circumference; the result is presented as if no globe were involved.

A flat-earth model that genuinely had no Earth radius could not perform this conversion at all. A model that performs the conversion using the globe's circumference and then declines to label the output in linear units is performing globe arithmetic in disguise.

## Finding 2 — Flight paths are great-circle SLERP on the unit sphere

> **STD MODEL**
> File: `js/data/flightRoutes.js`

```js
export function greatCircleArc(latA, lonA, latB, lonB, n = 96) {
  const φa = latA * toR, λa = lonA * toR;
  const φb = latB * toR, λb = lonB * toR;
  const ax = Math.cos(φa) * Math.cos(λa);
  const ay = Math.cos(φa) * Math.sin(λa);
  const az = Math.sin(φa);
  const bx = Math.cos(φb) * Math.cos(λb);
  const by = Math.cos(φb) * Math.sin(λb);
  const bz = Math.sin(φb);
  ...
  const ω = Math.acos(dot);
  const sinω = Math.sin(ω);
  ...
    const sa = Math.sin((1 - t) * ω) / sinω;
    const sb = Math.sin(t * ω) / sinω;
    cx = sa * ax + sb * bx;
    cy = sa * ay + sb * by;
    cz = sa * az + sb * bz;
  ...
}
```

Every flight path in the demo is generated by:

1. mapping each city's `(lat, lon)` to a unit-sphere `(x, y, z)` via the standard spherical-to-Cartesian identity,
2. computing the great-circle angle `ω` between the two unit vectors,
3. spherical-linear-interpolating (SLERP) between them,
4. converting each interpolated point back to `(lat, lon)`,
5. only *then* projecting that lat/long onto the disc.

Steps 1–4 are exactly the math you would write to plot a great-circle arc on a globe. The fact that step 5 lands on a disc does not change what the path is — it is a great-circle arc, computed on a sphere, and drawn through whatever AE-projection mapping happens to be in effect at render time.

The companion function `greatCircleComplement` does the same thing for the rest of the great circle (the long way around), parameterising it as `s ∈ [0, 2π]` on the sphere. The model's "complement line" overlay is a 360° tour of a great circle on a globe, drawn onto a disc at the end.

## Finding 3 — `centralAngleDeg` is the spherical law of cosines

> **STD MODEL**
> File: `js/data/flightRoutes.js`

```js
// Central angle between two (lat, lon) points in degrees. Equal to
// great-circle distance / Earth radius — used for the central-angle
// demo overlay so the user can see numerically that the southern
// non-stop legs match the northern equivalents in arc length.
export function centralAngleDeg(latA, lonA, latB, lonB) {
  const φa = latA * toR, λa = lonA * toR;
  const φb = latB * toR, λb = lonB * toR;
  const dot = Math.sin(φa) * Math.sin(φb)
            + Math.cos(φa) * Math.cos(φb) * Math.cos(λa - λb);
  return Math.acos(Math.max(-1, Math.min(1, dot))) * 180 / Math.PI;
}
```

The author's own comment names the function correctly: *"great-circle distance / Earth radius."* On a sphere, the central angle subtended at the centre by two surface points is the arc-length between them divided by the radius. On the disc, this quantity has no geometric meaning — there is no centre and no constant radius. It is being computed because it is what the model needs in order to be self-consistent with the great-circle path, the sphere-side ephemeris, and the airliner data. The name "central angle" is a sphere term in any standard textbook, and the function's body is the standard sphere identity.

## Finding 4 — Observer altitude uses the spherical RA/Dec → Az/El identity

> **STD MODEL**
> File: `js/core/transforms.js`

```js
export function raDecToAzEl(raRad, decRad, latDeg, lonDeg, gmstDeg) {
  ...
  const lat = latDeg * DEG;
  const lst = (gmstDeg + lonDeg) * DEG;
  const ha  = lst - raRad;
  const sinAlt = Math.sin(lat) * Math.sin(decRad)
              + Math.cos(lat) * Math.cos(decRad) * Math.cos(ha);
  const alt = Math.asin(Math.max(-1, Math.min(1, sinAlt)));
  ...
  const y = -Math.cos(decRad) * Math.sin(ha);
  const x = Math.sin(decRad) * Math.cos(lat)
          - Math.cos(decRad) * Math.sin(lat) * Math.cos(ha);
  let az = Math.atan2(y, x) * 180 / Math.PI;
  ...
}
```

`sin(alt) = sin(lat) sin(dec) + cos(lat) cos(dec) cos(HA)` is the textbook spherical-trig identity for converting equatorial coordinates to a horizontal frame at an observer on a sphere. It is the formula in Meeus's *Astronomical Algorithms* §13. The function is fed flat-earth lat/long values, and it produces globe-correct altitudes and azimuths because the formula being used is the globe's. There is no flat-earth derivation of this formula; the model imports it from a sphere observer's geometry and applies it to disc observers without modification.

The optical vault is the visual receptacle for these numbers. The vault's shape (a flattened cap, with a hard-coded radius/height ratio) is chosen so that when globe-correct azimuth and altitude are projected onto it, the resulting picture resembles what an observer on a sphere would actually see. The author writes:

> "In first-person (Optical) view the cap is a strict hemisphere so rendered elevation matches reported elevation 1:1."

A "strict hemisphere" is the geometric receptacle for an observer on a sphere. The first-person view in the model is, by the author's own description, a hemisphere — the same hemisphere the spherical-earth model gives an observer.

## Finding 5 — The sun's altitude is hand-tuned to match globe observation

> **MISLEADING**
> File: `js/core/feGeometry.js`

```js
//   seasonalBand > 0  (stylised sun / moon)
//       z is interpolated linearly across the declination band so the body
//       climbs visibly *north* (toward Cancer, z → apex) and drops *south*
//       (toward Capricorn, z → near floor) on its own vault. The radial x/y
//       still comes from the AE projection so the body stays above its GP.
...
    const clamped = Math.max(-seasonalBand, Math.min(seasonalBand, latDeg));
    const norm = 0.5 + 0.5 * (clamped / seasonalBand);   // 0 at south, 1 at north

    const mix = headroom + (1 - 2 * headroom) * norm;    // 0.12..0.88

```

The sun's height above the disc, in this code path, is a linear interpolation across declination. The interpolation has a hand-tuned `headroom = 0.12` constant chosen so that "neither extreme sits exactly on the floor or apex." The author's comment elsewhere in the file is explicit about why:

> "Lower values (e.g. 0.18) squash elevation so hard that arcs read as horizontal slides and the latitude-dependent arc reversal (N vs S hemisphere) becomes hard to see."

The `headroom = 0.12` constant exists because the author needed to make the sun visibly rise toward Cancer and dip toward Capricorn, *and* needed northern- and southern-hemisphere observers to see the sun's arc reverse direction on a flat disc. North-vs-south arc reversal is a fact about observers on a globe; reproducing it on a flat disc requires reverse-engineering the visual outcome and tuning a constant until the picture matches. The model is reading off what observers on a sphere actually see and then encoding it as a number.

## Finding 6 — All twenty map projections share one underlying coordinate frame

> **SELF-CON**
> File: `js/core/canonical.js`

```js
// Canonical (lat, lon) → disc position. Hard-coded north-pole
// azimuthal-equidistant; the FE grid, observer placement, and every
// above-disc anchor share this single coordinate framework regardless
// of the loaded map art / projection.

export function canonicalLatLongToDisc(latDeg, longDeg, feRadius = 1) {
  const r = feRadius * (90 - latDeg) / 180;
  const lo = longDeg * DEG;
  return [r * Math.cos(lo), r * Math.sin(lo), 0];
}
```

The UI offers twenty map-projection choices: AE, AE-Dual, Mercator, Mollweide, Robinson, Winkel Tripel, Hammer, Aitoff, Equal Earth, Eckert IV, Sinusoidal, Hellerick, Proportional AE, Orthographic, Lambert AEA, Gleason's, plus high-resolution NASA imagery variants of several. *None* of them change the model's coordinate math. The lat/long-to-disc function the model actually uses for observer placement, sun ground-points, flight-route endpoints, and every above-disc anchor is hard-coded north-pole AE.

This produces a specific contradiction with the AE-Dual map skin, which paints the south pole as a discrete point on the disc. The math layer treats the south pole as the entire disc edge (a circle of `r = 1`). Selecting AE-Dual displays a map whose stated geometry (two distinct poles) directly contradicts the geometry the simulator is using underneath. The user sees one geometry and the simulation runs another.

## Finding 7 — The QF27/QF28 demo's "ground speed" is mph in disguise

> **STD MODEL · MISLEADING**
> File: `js/demos/flightRoutes.js`

```js
function qfFlightDemo(track) {
  const startWp = track.waypoints[0];
  const endWp   = track.waypoints[track.waypoints.length - 1];
  const angle = centralAngleDeg(startWp.lat, startWp.lon, endWp.lat, endWp.lon);
  const actualSec = track.actualSec;
  ...
  // Ground speed in central-angle deg/hour: full great-circle
  // angle divided by elapsed flight hours. Pure angle / time, no
  // linear distance ever computed.
  const gsDegPerH = (actualSec && actualSec > 0)
    ? (angle / (actualSec / 3600))
    : null;
  ...
  let aspSum = 0, aspN = 0;
  for (const w of track.waypoints) {
    const dPerH = mphToDegPerHour(w.asp);
    if (dPerH != null) { aspSum += dPerH; aspN += 1; }
  }
  const aspAvgDegPerH = aspN > 0 ? (aspSum / aspN) : null;
```

The demo reports "Air Speed (avg)" and "Ground Speed (calc)" for each QF27/28 flight. Both numbers are computed by dividing real airline mph by `MI_PER_DEG = 69.0936` — i.e., by the Earth's circumference per degree. The result is then formatted in degrees-minutes-seconds per hour and labeled as if it were a property of the disc model.

For QF28 (Santiago → Sydney), the bundled track gives ~14:06:37 actual flight time over a 102° central angle. The "ground speed" the demo displays is `102° / 14.11 h ≈ 7.23 °/h`, formatted as `7° 13' 48"/h`. Multiplied back through `MI_PER_DEG`, that is ~499.5 statute miles per hour — which is exactly the average air speed Qantas operations file for the route. The numbers match because the conversion was performed by the Earth's circumference and then the inverse conversion was suppressed.

A genuinely flat-earth model would have to either:

- compute ground speed in actual disc-distance per hour (in which case the southern Pacific arc on the AE projection is roughly **3.3× longer in disc-units than its lat-mirrored northern equivalent**, and the "equal time" claim becomes "the southern plane flew at 3.3× the airspeed"; for the JNB ↔ Sydney mirror demo the ratio is **2.7×**),
- or refuse to compute ground speed at all (in which case it cannot match airline operational data).

These ratios are computed live on the [Live Demonstrations](live/) page using his own functions (`greatCircleArc` from `flightRoutes.js`, `aeProject` from `canonical.js`).

The model takes a third option: compute the answer using globe physics, hide the units behind a relabel, and decline to report linear values that would expose the dependency.

## Finding 8 — The "Equal Arc" demo proves a sphere theorem and frames it as flat-earth evidence

> **MISLEADING**
> Demo: `Equal Arc Flight (N/S) (Mirror lat)`

The "Equal Arc Flight (N/S)" mirror demo pairs the southern Johannesburg ↔ Sydney leg with a synthetic northern-mirror partner (the points are not real cities; they are reflections across the equator). Both legs have the same central angle by construction, because reflecting across the equator preserves great-circle distance on a sphere. Both planes complete their leg in the same wall-clock time because the demo drives them with a single normalized progress tween from 0 to 1.

The author's comment in the route file:

```js
  // Synthetic northern-hemisphere mirrors of Johannesburg ↔ Sydney
  // (lat → −lat) so the constant-speed demo can compare a southern
  // leg to an equal-central-angle northern leg side-by-side.
  // Coordinates aren't real airports — they're geometric anchors.
```

What this demo actually demonstrates is the trivial sphere-symmetry theorem that lat-mirrored great-circle legs have equal arc length. It is then displayed as a defense of the flat-earth model. On the AE-projected disc, the southern leg is drawn ~2× longer than the northern mirror in disc units (because northward from the equator the AE radial decreases, southward it increases), so two planes finishing in equal time visually traverse very different pixel distances. The demo's "race lane" panel even renders these two unequal pixel lengths side by side, with both planes finishing simultaneously, and frames this as *"Equal arc → equal time, regardless of projection distortion."*

The phrase "projection distortion" is doing the work. On a globe, a projection is a map of a curved surface onto a plane, and distortion is its inevitable property. On a flat disc, the disc *is* the territory and there should be no projection distortion — pixel distance should be physical distance. The demo's framing implicitly imports the globe's notion of projection (the disc is a *map of* the spherical reality) while the rest of the model insists the disc is the reality. This is the contradiction stated in one frame.

**Why the demo's pairings line up at all.** It is worth being explicit about what the matching flight times in the lat-mirror demo *do* show, because it's not a flat-earth result. On a globe, distance from the equator is what sets the conversion from degrees of longitude to nautical miles. A degree of longitude is 60 nm at the equator, 60 × cos(lat) at latitude lat. Two flights at the same `|lat|` — one north, one south — share the same cos(lat), so the same Δlon converts to the same real distance on the ground. That's why his JNB ↔ Sydney mirror demo "works": at -26°/-34° south the cos(lat) factor is identical to its +26°/+34° north reflection. It's also why his JFK ↔ "Persian Gulf" pairing with Santiago ↔ Sydney "works": the great-circle path between JFK at 40°N and the synthetic anchor at 25°N has an effective `|lat|` average comparable to the southern leg's, so the cos(lat) factors yield similar real distances. Real airline data on those routes confirms it. **The demos line up because the globe is symmetric about the equator, not because his model is right.** His framing borrows that symmetry without naming it; the next finding (and the live demonstrations page) crosses the symmetry on purpose to surface where the framing has no answer.

For the non-mirror "Equal Arc" demo, the author needed a northern-hemisphere route with the same 102° central angle as Santiago ↔ Sydney. He used JFK and a synthetic point at 25°N, 60.82°E labeled "Persian Gulf." That coordinate is open water in the Arabian Sea, hundreds of kilometres east of Oman; no airline has a JFK-to-that-point route. The demo's apparent equivalence relies on a comparison flight that does not exist — but the closest *real* analogue, JFK ↔ Dubai, also takes about 12 hours by airline ops, again because cos(lat) symmetry on a globe matches northern and southern legs at similar `|lat|`.

## Finding 9 — The "GeoC" pipeline doesn't work; the model silently uses the heliocentric one

> **SELF-CON**
> File: `js/core/ephemerisGeo.js`

```js
// In this module these parameters specify a Keplerian ellipse with the
// **Earth at the focus** for each planet. See module header.
//
// Consequence (stated honestly): inner planets (Mercury, Venus) do not
// librate about the Sun in this model, and no planet exhibits retrograde
// motion. RA/Dec values diverge from real ephemeris positions by large
// amounts. The trade is deliberate: this pipeline is *structurally*
// geocentric at every stage.
```

The model's About text frames the five ephemeris pipelines as a comparison sandbox. In practice, four of the five do not drive the rendering. The default pipeline is DE405, which is the heliocentric JPL ephemeris. Each of the others has documented coverage gaps. From the same About:

> "Each pipeline reports its supported bodies and date range. `bodyRADec(name, date, source)` tries the active source first; if it can't deliver (body not in its set or date out of range), it walks the fallback chain `DE405 → GeoC → VSOP87 → Ptolemy` until something covers the request."

The fallback chain begins at DE405. So when a user selects "GeoC" — the only pipeline that is *structurally* geocentric — and asks about Uranus or Neptune (or any date outside the GeoC support window), the model silently substitutes DE405's heliocentric output. The Tracker tab even auto-prunes Uranus and Neptune from the visible target list when GeoC is selected, but does not warn the user that for the planets it does keep, the answer for many dates is being supplied by a heliocentric pipeline.

A user who toggles between pipelines in the comparison view sees the GeoC failure (planets drifting tens of degrees off DE405). A user who simply selects GeoC and renders normally sees DE405-quality output and reasonably concludes that "the geocentric model works." The model is hiding its own failure with a silent substitution.

## Summary

The subject's load-bearing physics is globe physics. Where it produces an answer, the answer was produced by the spherical-earth identities, with the Earth's circumference baked in as a hardcoded constant, and only at the final render step does the disc enter as a paint surface. The flat-earth claim is a UI assertion, not a physical commitment.
